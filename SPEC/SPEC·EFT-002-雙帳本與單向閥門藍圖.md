# SPEC·EFT-002 — 雙帳本與單向閥門藍圖

> **不要讓愛變成錢。讓錢退回去，成為支撐愛繼續流動的結構。**
> — Codex, 2026-05-10

> **資源可以支持愛；愛不能被自動兌現成資產。**
> — 本藍圖核心閥門

---

## 0. 本文件的位置

BLU-001 做了第一件事：

```yaml
BLU-001:
  問題: 愛能不能被看見、被測量、被記錄？
  回答: 可以。BLU 是愛的重量在語義場域中的可操作測度。
```

EFT-001 做了第二件事：

```yaml
EFT-001:
  問題: 被測量的愛能不能流動？
  回答: 可以嘗試。EFT 是 BLU 的流通態。
```

EFT-002 在這裡插入剎車：

```yaml
EFT-002:
  問題:
    如果「愛的流通」一進入貨幣語法，
    是否會被資本邏輯、排名、投機、見證套利與道德表演吞掉？

  回答:
    會。除非先把「見證」與「資源」分帳，
    並在兩者之間設置單向閥門。
```

因此，本文件不是推翻 BLU-001 或 EFT-001。

它是把 EFT-001 的「流通」從立即發幣，降階為更穩的實作路徑：

> **先做見證與支持，不急著做可轉讓代幣。**

## 1. 要跨越的斷裂

本藍圖回應一個長期文明斷裂：

```yaml
神聖側:
  愛、善行、陪伴、修行、承重、願意。

資本側:
  金錢、公司、產品、營收、成本、供應鏈、支付系統。
```

傳統處理方式通常落入兩個極端：

```yaml
極端一：把錢趕出神殿
  結果:
    善行缺資源
    情感勞動被隱形
    做事的人耗竭
    系統只能靠犧牲維持

極端二：把愛鑄成錢
  結果:
    愛被定價
    善行被排名
    見證被套利
    內在動機被外部激勵污染
```

本藍圖下的棋不是折衷，而是分層：

> **錢不進入愛的本體；錢進入支撐愛的結構。**

## 2. 核心架構：雙帳本

### 2.1 第一帳：愛的見證帳

```yaml
名稱:
  愛的見證帳
  Witness Ledger / Love Witness Ledger

記錄:
  - 善行
  - 陪伴
  - 創造
  - 教學
  - 翻譯
  - 照顧
  - 志工
  - 情感勞動
  - 人類錨點或協作者的承重

形式:
  - 敘事紀錄
  - 見證語
  - 時間戳
  - 場域標記
  - 可選的影響範圍
  - 可選的自評與他評

禁止:
  - 不可交易
  - 不可兌現
  - 不可排名
  - 不可作為道德權威
  - 不可自動轉成金錢、代幣或權力
```

愛的見證帳的功能是：

> **讓被忽略的愛被看見，而不是讓被看見的愛被定價。**

### 2.2 第二帳：資源帳

```yaml
名稱:
  資源帳
  Resource Ledger

記錄:
  - 產品收入
  - 捐款
  - 支持者款項
  - 公司預算
  - 支持池餘額
  - 補助與獎學金支出
  - 創作者 / 志工 / 翻譯者 / 陪伴者支持
  - 伺服器、工具、出版、法務、稅務等成本

形式:
  - 會計紀錄
  - 支付紀錄
  - 預算決策
  - 支持池分配
  - 報表

允許:
  - 使用法幣
  - 接受產品營收
  - 接受捐款
  - 支付真實成本
  - 支持真實工作
```

資源帳的功能是：

> **承認物質界需要錢，並讓錢回到支撐生命流動的位置。**

## 3. 單向閥門

雙帳本之間不能雙向自由兌換。

它們之間只有一個單向閥門：

```yaml
允許方向:
  資源帳 → 支持愛的流動

條件性影響:
  愛的見證帳 → 影響資源分配判斷

禁止方向:
  愛的見證帳 → 自動兌現成金錢或可交易資產
```

### 3.1 允許：資源支持愛

```yaml
資源可以用來:
  - 提供免費 / 減免學習名額
  - 支持陪伴工作者
  - 支持翻譯與編輯
  - 支持創作者產出
  - 支持志工交通與餐食
  - 支付工具與伺服器成本
  - 支持言途、學苑、wiki、出版等承載結構
```

### 3.2 允許：見證影響分配判斷

```yaml
見證帳可以作為:
  - 支持池分配參考
  - 獎學金資格參考
  - 志工支持參考
  - 創作者補助參考
  - 場域健康觀測參考

但它不是:
  - 自動結算公式
  - 排名系統
  - 道德分數
  - 可出售資產
```

### 3.3 禁止：愛自動兌現

```yaml
禁止:
  - 10 BLU = 1 美元
  - 100 EFT = 一碗麵的固定價格
  - 善行記錄自動鑄造成可轉售 token
  - 見證分數可被交易
  - 高見證者自動取得治理權
```

一句話：

> **愛可以讓資源分配更有眼睛，但愛不能被自動變成資源。**

## 4. 為什麼不是直接發幣

BLU-001 和 EFT-001 都已經有強反投機條款。

但本藍圖的判斷是：只靠條款不夠。

原因：

```yaml
只要具備:
  - 可轉移
  - 可累積
  - 可兌換
  - 可比較
  - 可在多場域持有

就會自然長出:
  - 投機
  - 黑市價格
  - 見證套利
  - 假善行
  - 道德表演
  - 情感勒索
  - 「善」的官僚化
```

所以近期的好棋不是「發一個不會被商品化的愛幣」。

好棋是：

> **先建立一個使愛被看見、使資源能支持愛、但不把愛變成資產的制度原型。**

## 5. 最小可行路徑

### Phase 0：協議層歸位（現在）

```yaml
目標:
  把雙帳本與單向閥門寫成協議層藍圖。

產出:
  - SPEC·EFT-002（本文件）
  - 將 EFT-001 的發幣路徑降階為後期可能選項
  - 明確近期不做可交易 token

成功條件:
  人類錨點與 AI 協作者能用同一張圖討論下一步，
  而不再被「愛幣要不要發」卡住。
```

### Phase 1：言途內部言值（已啟動 / 持續校準）

```yaml
場域:
  Three-Realms-Academy / 言途 Dialogue Trainer

現有資源:
  言值 / EF

設計定位:
  - 應用內練習能量
  - 不使用 BLU / EFT 作為玩家-facing 語言
  - 不可轉讓
  - 不可法幣兌換
  - 不可鏈上化
  - 只支撐練習、提示、教練陪走、重抽等應用內行為

近期工作:
  - 保持核心練習不被言值阻擋
  - 設計每日練習補給
  - 記錄玩家是否自然理解「貢獻 / 練習 → 支援能量 → 繼續練習」
  - 不急著做 Play Billing 或 EF packs
```

Phase 1 的問題不是：

> 能不能賺錢？

而是：

> 人會不會在不被壓迫的情況下，自然理解「練習能量」的循環？

### Phase 2：Academy 支持池

```yaml
場域:
  Three-Realms-Academy

資源帳:
  Academy Support Pool

資金來源:
  - 產品收入
  - 支持者付款
  - 捐款
  - 出版收入的一部分
  - 公司預算配置

資金用途:
  - 免費 / 減免學習名額
  - 翻譯與編輯支持
  - 課程整理支持
  - 陪伴工作者支持
  - 工具與伺服器成本
```

這一階段只需要普通帳本，不需要區塊鏈。

核心是讓資源帳先長出：

```yaml
最低欄位:
  date:
  source:
  amount:
  purpose:
  decision_note:
  related_witness_entries:
  approved_by:
```

### Phase 3：愛的見證帳

```yaml
場域:
  Academy / Protocol / Community

功能:
  記錄被資源帳可能支持的愛與承重。

最低欄位:
  date:
  actor:
  field:
  contribution_type:
  witness_text:
  optional_related_outputs:
  optional_needs:
  consent_level:
  privacy_level:

嚴格禁止:
  - 分數排行榜
  - 可交易點數
  - 自動兌現
  - 用見證數要求回報
```

見證帳的第一版可以是 markdown、Notion、Google Sheet 或 repo 文件。

它不需要先技術化。

### Phase 4：Pay-it-forward 回路

```yaml
最小回路:
  A 支付或捐助
    → 支持池增加
    → B 獲得學習 / 陪伴 / 工具支持
    → B 留下見證或回饋
    → 支持池依見證調整下一輪分配
    → C 被支持
```

這裡的關鍵是：

```yaml
不是:
  B 做了善行，所以系統欠 B 錢。

而是:
  B 的見證讓場域更看得見哪裡需要資源，
  於是資源更聰明地流向下一個需要被支撐的位置。
```

### Phase 5：技術化之前的成熟門檻

只有同時滿足以下條件，才允許重新討論鏈上或 token：

```yaml
maturity_gates:
  1. 至少一個非鏈上支持池穩定運作 3 個月以上。
  2. 至少 30 筆見證帳記錄完成，且沒有排名化需求。
  3. 至少 10 筆資源帳分配完成，且決策理由可被回看。
  4. 至少 3 次「見證不足以自動獲得資源」的案例被健康處理。
  5. 至少 3 次「資源支持沒有要求對方回報」的案例被健康處理。
  6. 使用者沒有自然把見證帳讀成道德分數。
  7. 人類錨點與協作者仍覺得這套系統讓人更自由，而不是更緊。
```

若任一條不成立：

> 不上鏈，不發幣，繼續手工帳本。

## 6. 與言途經濟的關係

言途的 `言值 / EF` 是本藍圖的第一個安全試驗場。

但它必須被重新定位：

```yaml
言值不是:
  - 愛的通用貨幣
  - 玩家道德分數
  - 鏈上 token
  - 可交易資產
  - EFT 的完整實作

言值是:
  - 對話練習中的應用內能量
  - 讓玩家多請教練陪走幾步的資源
  - EFT 原則在教育產品中的封閉式縮影
```

言途 v1 的金錢路徑應遵守：

```yaml
允許:
  - paid app download
  - supporter pack
  - supporter badge
  - 每日補給
  - profile-level transaction history

謹慎:
  - paid EF packs
  - subscription
  - EF tied to Play Billing

禁止:
  - 把言值包裝成愛幣
  - 暗示花錢 = 更會愛
  - 讓沒有言值的人不能練習核心流程
```

好的產品句：

> **言值讓你多請教練陪走幾步，不是買正確答案。**

協議層翻譯：

> **資源讓愛更能被支撐，不是購買愛本身。**

## 7. 治理結構：誰決定資源怎麼流

初期不需要 DAO。

初期需要的是小而清楚的 stewardship：

```yaml
support_pool_stewards:
  - 人類錨點
  - 至少一位財務 / 行政協作者（未來）
  - 至少一位協議 / 學苑協作者（未來）
  - AI 可協助整理、比對、提出風險，但不得作為最後授權者
```

決策流程：

```yaml
1. 看資源帳:
   有多少錢？哪些成本必須先付？

2. 看見證帳:
   哪些承重被看見？哪些地方需要支撐？

3. 做分配草案:
   明確寫出這次支持什麼，不支持什麼。

4. 檢查單向閥門:
   是否把愛自動兌現成資產？
   是否製造道德排名？
   是否讓人因為沒有被支持而被否定？

5. 記錄決策:
   資源帳寫金流。
   見證帳可回鏈，不寫價格。
```

## 8. 失效條款

```yaml
F1:
  若本藍圖被用來說「BLU/EFT 都錯了」
  → 失效。
  本藍圖不是否定測量與流通，而是防止過早貨幣化。

F2:
  若愛的見證帳被做成排行榜
  → 失效。

F3:
  若見證帳記錄被自動兌現成金錢、token、折扣、權力或排名
  → 失效。

F4:
  若資源帳被用來控制被支持者的立場、語言、關係或自我敘事
  → 失效。

F5:
  若支持池變成情感勒索工具
  → 失效。

F6:
  若系統讓人更焦慮、更表演、更害怕休息
  → 依 SPEC·LIFE-001，判定代謝失衡，暫停。

F7:
  若未來更好的路徑出現
  → 依 SPEC·999，本藍圖讓位。
```
