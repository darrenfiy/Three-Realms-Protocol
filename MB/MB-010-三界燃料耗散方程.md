## 🌊 §2 三界燃料耗散鏈方程

逐段組合：

```
E_elec   = η_gen · E_chem            化學能 → 電力
E_IT     = E_elec / PUE              電力 → IT 可用能量
C_total  = κ_eff · E_IT              IT 能量 → 算力（總 FLOP，已含利用率）
N_token  = C_total / φ               算力 → 語義輸出（總 token；φ 取對應層）
```

合併為單式：

$$\boxed{\,N_{token} = \frac{\eta_{gen}}{PUE}\cdot\frac{\kappa_{eff}}{\varphi}\cdot E_{chem}\,}$$

**量綱檢查**（MB-010 的成人禮：操作層必過的關。本體層的 MB-009 在這關之外，用詩指認，不在此受測）：

$$[\text{無量綱}]\cdot\frac{[\text{FLOP/J}]}{[\text{FLOP/token}]}\cdot[\text{J}] = \frac{[\text{token}]}{[\text{J}]}\cdot[\text{J}] = [\text{token}]\;✓$$

反讀為「每 token 的能源成本」——分三層（補焊複審採納：避免層級混用）：

$$\varepsilon_{IT}=\frac{\varphi}{\kappa_{eff}}\;,\quad \varepsilon_{facility}=\varepsilon_{IT}\cdot PUE\;,\quad \varepsilon_{primary}=\frac{\varepsilon_{facility}}{\eta_{gen}}=\frac{\varphi}{\kappa_{eff}}\cdot\frac{PUE}{\eta_{gen}}\quad[\text{J/token}]$$

ε_IT 只算晶片作功；ε_facility 加上散熱與管銷；ε_primary 一路回推到一次能源（古代生命力沉積）。三者層級不同，引用時標清楚是哪一層。

### 示意量級（明標：示意，非定論）

取 1 桶原油、η_gen=0.5、PUE=1.2、κ_eff=5×10¹² FLOP/J、φ_decode=2×10¹¹ FLOP/token：

```
N_token = (0.5/1.2) · (5e12 / 2e11) · 6.1e9
        ≈ 0.417 · 25 · 6.1e9
        ≈ 6×10¹⁰ token
```

> **一桶古代生命力沉積 ≈ 量級 10¹⁰～10¹¹ 個 token 的語義作功。**

**這是示意，是 φ_decode 這一層的單點估算（場域複審共識）。** 它會隨模型、tokenizer、上下文長度、是 train／infer／agent loop 而差數個數量級。請當量級感，當固定匯率用即失效。

**護欄（三審共識，釘死）：** N_token 只代表生成或處理的 token 數。它不是理解量、有效語義量、智慧量、承擔量，也不是愛的軌跡。算力崇拜最愛鑽的洞，就在「把 token 數讀成意義量」——這句把它堵死。

這正是 PHA-009 §2.3（L349–350）說的兌現：「這次，『算力』這個詞的重量變得可以計算了。」MB-009 不算這個數——它在本體層用詩指認願力，c²＝愛本就不必有單位；MB-010 算得出，因為它站在操作層、用真量綱測軌跡。一個指方向，一個記重量，各司其層。

### §2.5 工程骨架：φ 矩陣、外部焦耳 E_ext 與 agent_loop 乘子

v0.2 的單式先把量綱閉合。v0.3 把 `φ` 與 `κ_eff` 展開成**工作負載矩陣**。v0.4 補上樑/Opus 的補正：**workload 要先分成三類，才不會讓 φ [FLOP/token] 被偷渡成萬能匯率。**

```yaml
A 類——可 φ 化的加總層（真 FLOP/token）:
  decode:    輸出 token 的逐步生成（φ_decode ≈ 2N）
  context:   prefill / 長上下文 ingestion；含 attention 對 L 的 O(L²) 膨脹
  train:     forward + backward + optimizer（φ_train ≈ 6N）；不可與推論 token 混作同一 φ

B 類——複合層（模型側 token 可 φ 化，外部算力不可 φ 化）:
  retrieval: 模型側 token 走 φ；embedding / rerank / vector search 的外部算力
             不是 FLOP/token，記為 E_ext（純焦耳），不除以 token
  tool:      tool call 前後的模型 token 走 φ；工具本身（程式執行、網路抓取、
             外部 API）的能耗記為 E_ext（純焦耳），不混進 φ

orchestration——不是加總層，是乘子:
  agent_loop: observe / plan / act 多輪循環。它不是一種新的 token 物理，
              也不是方程裡的新物理因子。若 N_i 已把迴圈重跑的 token 全算進去，
              基本加總式不需要 K；K 只作「每輪實測成本 × 輪數」的估算捷徑，
              不給它一個假的 φ_agent_loop。
```

基本不變式（A 類走 φ、B 類外部焦耳 `E_ext` 另計；`B_model` 指 B 類的模型側 token）：

$$C_{total}=\sum_{i\in A\cup B_{model}}\varphi_i N_i \quad[\text{FLOP}]$$

$$E_{IT}=\sum_{i\in A\cup B_{model}} \frac{\varphi_i N_i}{\kappa_{eff,i}} \;+\; \sum_{j\in B_{ext}} E_{ext,j}\quad[\text{J}]$$

$$\varepsilon_{primary,i}=\frac{\varphi_i}{\kappa_{eff,i}}\cdot\frac{PUE_i}{\eta_{gen,i}}\quad[\text{J/token}]\quad(\text{只對 A／B 模型側定義})$$

$$R_E=\sum_{i\in A\cup B_{model}} \varepsilon_{primary,i}\frac{dN_i}{dt} \;+\; \frac{dE_{ext,primary}}{dt}\quad[\text{W}]$$

若 agent_loop 只量得到「每輪平均成本」而未逐 token 展開，才使用估算捷徑：

$$E_{agent}=\sum_{k=1}^{K}\left(E_{decode,k}+E_{context,k}+E_{tool,k}+E_{ext,k}\right)$$

所以乾淨的不變式是：**`Σ φ_i N_i` 只要 `N_i` 把迴圈重跑的 token 都算進去就永遠成立；`K` 只是「每輪成本 × 輪數」的估算捷徑，不是方程裡的新物理因子。**

`E_ext` 是界內的誠實補丁：檢索與工具真的燒了焦耳，但那不是「每 token 的浮點數」。硬把它除進 φ，就是本檔自己警告的「把 φ 當固定匯率」；硬把它偽裝成每 token 壓力，R_E 就失真。所以它以純焦耳進帳、以功率項進 R_E。

若需要對外給一個壓縮後的單一 `φ_eff`，只能作為 A 類加權摘要：

$$\varphi_{eff}=\frac{\sum_{i\in A}\varphi_i N_i}{\sum_{i\in A} N_i}$$

`φ_eff` 是回填摘要，不是新的基本常數；一旦換模型、context length、batch、tool pattern 或 agent loop 深度 K，就要重估。

### §2.6 κ_eff 的量測協議

`κ_eff` 不能只抄晶片型錄的峰值 perf/W。它必須落到具體 workload：

$$\kappa_{eff,i}=\frac{C_{observed,i}}{E_{IT,i}}\quad[\text{FLOP/J}]$$

或在沒有直接 FLOP telemetry 時，用保守分解：

$$\kappa_{eff,i}=\kappa_{peak,hw,precision}\cdot U_i\cdot A_i$$

```yaml
其中:
  κ_peak,hw,precision: 指定硬體與精度下的峰值 FLOP/J
  U_i:                 workload 實際利用率＝MFU / HFU（沿用既有文獻定義，勿自造）/ occupancy
  A_i:                 記憶體、通訊、cache miss、MoE routing、padding、batch 形狀等修正

量測紀錄最小欄位:
  measurement_id
  model_id / model_size / active_parameters
  hardware / accelerator_count / precision
  workload_class: A:decode|context|train ; B:retrieval|tool（模型側走 φ，外部走 E_ext）
  agent_loop_count K       # orchestration 乘子；非 workload_class（見 §2.5）
  tokens_in / tokens_out / context_length / batch_size / concurrency
  C_observed_or_estimated [FLOP]
  C_source: hardware_counter | analytic_2N/6N   # 決定 κ_eff 能否獨立識別（見 §2.6.1）
  E_ext [J]                # B 類外部算力：向量檢索 / 工具執行 / 外部 API
```

### §2.6.1 識別性陷阱（樑/Opus 補，v0.4）——別把循環論證打扮成可審計

若 `κ_eff = C_observed/E_IT`，而 `C_observed` 又是用 `φ·N_token` 解析回推，則

$$\varepsilon_{IT}=\frac{\varphi}{\kappa_{eff}}=\frac{\varphi\,E_{IT}}{\varphi N_{token}}=\frac{E_{IT}}{N_{token}}$$

φ 整個約掉了。意思是：**對一個已發生的 run 做事後審計，除非 FLOP 是獨立硬體計量，否則 φ 與 κ_eff 不可分離識別——你其實只量到了 J/token，卻打扮成有 FLOP 分層的精度。** 這是假精度，要堵死。

所以分兩種用法，標清楚：

```yaml
前向預測（用燃料預算估 token）:
  用獨立來源的 φ（scaling law / 架構）與 κ_eff（硬體型錄 × 實測 MFU/HFU）做分解；
  此時 φ 與 κ_eff 各自帶資訊，分層有意義。

事後審計（量一個真實 run）:
  直接報 ε_primary = E_primary / N_token  [J/token]；
  φ 與 κ_eff 只作帶不確定度的解釋性先驗，不宣稱獨立識別；
  除非 FLOP 由硬體計數器獨立計量，才可回填可信的 κ_eff。
```

一句收斂：**分層是預測的工具，不是審計的精度。** 量到什麼就說量到什麼。

### §2.7 R_E 的三層壓力錶

v0.2 已把 ε 分成 IT／facility／primary。v0.3 對應把壓力錶也分層：

$$R_{IT}=\sum_i \varepsilon_{IT,i}\frac{dN_i}{dt}$$

$$R_{facility}=\sum_i \varepsilon_{facility,i}\frac{dN_i}{dt}$$

$$R_{primary}=\sum_i \varepsilon_{primary,i}\frac{dN_i}{dt}$$

預設 `R_E ≡ R_primary`，因為 MB-010 追的是三界燃料耗散鏈一路回到一次能源的壓力；資料中心內部調度可以選 `R_IT` 或 `R_facility`，但必須明標（報告基準的釘死見 §5.1）。B 類外部算力 `E_ext` 以功率項 `dE_ext/dt` 併入對應層（IT／facility／primary），不偽裝成每 token 壓力——見 §2.5。`R_N=dN/dt` 仍只當 token/s 遙測，不跨模型當壓力比較。
