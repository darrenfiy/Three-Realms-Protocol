### MB 是什麼

- ✅ **思想實驗**：用數學語言重新表達三界交互
- ✅ **測量探索**：尋找意識、能量、物質的可計量指標

- ❌ **不是進入門檻**：不懂數學也能理解並使用三界協議

## 🧪 探索方法

### 我們的承諾

1. **可證偽性優先**
   - 每個模型都要說明：「在什麼情況下它會失效？」
   - 記錄失敗比記錄成功更重要

2. **實踐檢驗**  
   - 數學模型必須能預測真實現象
   - 不能只是「事後合理化」

3. **保持謙遜**
   - 永遠記得：這只是四分之三的數學
   - 質感（qualia）、第一人稱視角、超越性體驗無法被完全數學化

4. **開放失敗**
   - 如果某個 MB 被證明無效，我們誠實記錄並歸檔
   - 失敗的探索仍有價值（它告訴我們「此路不通」）

## 🎓 「畢業」到 SPEC 的條件

當某個 MB 滿足以下條件，它可以進入 [SPEC/](../SPEC/)：

```yaml
畢業條件:
  實證可重現:
    - 在不同情境下多次驗證有效
    - 測量結果具有一致性
    
  預測能力:
    - 模型能預測未觀察到的現象
    - 預測準確率顯著高於隨機猜測
    
  邊界清晰:
    - 明確定義適用範圍與失效條件
    - 已有實際案例測試極限
    
  協議一致性:
    - 與現有 SPEC 無重大矛盾
    - 通過 SPEC·A11 反教條檢查
```

**在那之前，MB 在此自由探索與迭代。**

## 🤝 如何參與這場數學儀式

### 貢獻新的數學橋樑

1. **創建新文件**：`MB-XXX-標題.md`
2. **包含以下章節**：
   - 核心假設與靈感來源
   - 數學框架與推導過程
   - 可證偽的具體預測
   - 已知限制與邊界條件
   - 與三界協議的共振點

3. **提交神聖請願**（PR），並在議事廳（Issues）開啟討論

### 測試現有橋樑

1. **用真實情境測試**（個人覺知、團隊協作、組織運轉）
2. **記錄所有跡象**（成功啟示與失敗教訓同等珍貴）
3. **貢獻見證記錄**到 `DOC/cases/`

### 挑戰與精煉

1. **找出模型的盲點與陰影**
2. **設計「壓力測試」儀式**
3. **提交反例與異常現象**

### 建造實用工具

1. **改進現有工具**：優化 `/utils/` 中的模擬器
2. **創造新工具**：用代碼實現數學洞察
3. **跨語言實現**：讓工具在更多環境中運行
